# Spec: single-button-timing

> 单键时机游戏 — 守序者与闯入者

## 用户故事

作为玩家，我想在浏览器里玩一个「单键时机」小游戏：选一个角色（守序者或闯入者），在正确时机按下按键，体验「秩序 vs 混乱」碰撞的幽默与温馨。

- 守序者路径：我是一个爱整洁的小生物，要把被闯入者弄乱的东西归位。在正确时机按键 = 东西"啪"一声归位，爽；按错 = 东西又飞了，滑稽但不惩罚。
- 闯入者路径：我是一个乱糟糟的不搭界访客，想在这个整齐世界里别打乱想融入。在正确时机按键 = 成功伪装/不碰乱东西；按错 = 又碰乱了，搞笑。
- 两条路径最终汇合到「互相理解」的温馨结局。

## 验收标准（agent 可执行）

1. **核心时机判定**：存在纯函数 `judgeTiming(pressTime, windowCenter, windowHalf)`，返回 `{result, delta}`，覆盖 hit/early/late/miss 全分支。
   - 验证命令：`npx vitest run src/game/timing.test.js`
   - 断言：Vitest 输出全绿，4 个分支（hit/early/late/边界）均通过。

2. **双角色入口**：游戏开始界面显示两个角色选择按钮，点击后进入对应路径。
   - 验证命令：`npx playwright test e2e/character-select.spec.js`
   - 断言：Playwright 截图显示两个可点击区域；点击后 `window.__gameState.character` 为 `'order'` 或 `'chaos'`。

3. **难度递增**：随着分数增加，时机窗口缩小或目标移动速度加快。
   - 验证命令：`npx playwright test e2e/difficulty.spec.js`
   - 断言：testMode 下读取 `window.__gameState.level >= 1`，连续命中 5 次后 `window.__gameState.targetHalfWidth` 相比初始值减小 或 `speed` 增加。

4. **testMode 确定性**：`?testMode=1` 下游戏循环可脚本化断言，不依赖真实时间。
   - 验证命令：`npx playwright test e2e/testmode.spec.js`
   - 断言：固定 seed 下连续两次运行，`window.__gameState` 的 hit/miss/score 序列完全一致。

## In-scope

- 单键时机核心机制（命中/早/晚/错过判定）
- 双角色入口选择（守序者 / 闯入者）
- 成功/失败反馈（视觉 + 音效占位）
- 计分系统
- 重开机制
- 难度递增（窗口缩小 或 速度加快）
- 键盘 + 触摸双输入

- PWA 支持（manifest + viewport + 全屏）
- testMode 确定性测试钩子

## Out-of-scope

- 多机制（只做单键时机一个机制）
- 关卡编辑器
- 排行榜服务端 / 存档后端
- 在线多人

## 设计

### 视觉风格

- **秩序几何 vs 有机暖色**：
  - 守序者世界：几何对称、冷色系（蓝/青）、精确网格背景
  - 闯入者元素：有机曲线、暖色入侵（橙/黄）、打破网格
  - 配色 ≤4 色：深蓝（#1a1a2e，背景）、青色（#4ecdc4，秩序）、暖橙（#ff6b6b，混乱）、白（#ffffff，文字/高亮）
- **风格**：化简笔，不追求精细。角色用简单几何形状 + 少量特征（守序者=方头方脑+眼镜/尺子；闯入者=圆滚滚+羽毛/触角）
- **核心意象**：书桌/床铺/衣柜作为被归位的对象（守序者路径=东西飞回柜子；闯入者路径=东西差点被打乱）

### 音频气质

- **守序者世界**：整齐节拍、简约重复旋律（类似节拍器滴答）
- **闯入者元素**：不规则节奏、即兴感、温暖不和谐音
- **实现**：Web Audio API 合成（Oscillator + GainNode），T7 打磨阶段调音色
- **音频初始化**：只在用户首次按键/触摸后初始化（浏览器自动播放策略）

### 交互设计

- **单键**：空格键（桌面）/ 点击或触摸屏幕（手机）
- **触摸友好**：可点击目标 ≥44px、安全区域 env(safe-area-inset-*)、横竖屏自适应
- **反馈**：
  - 命中："啪"音效 + 东西归位动画 + 分数飘字
  - 早/晚：滑稽音效 + 东西飞走/晃动摇摆 + 不扣命（热闹不惩罚）
  - 错过（窗口飘过未按）：东西自己飞走，轻微"唉"声

### 屏幕流向

```
[标题画面] → [角色选择] → [游戏循环：目标移动→按键判定→反馈→下一个]
                ↓
         [守序者路径] / [闯入者路径]
                ↓
         [结局画面：两角色相遇/理解]
                ↓
         [重开 / 换角色]
```

## 技术任务

1. 实现 `judgeTiming` 纯函数 + Vitest 单测（T5）
2. 角色选择界面（Canvas 或 DOM 覆盖层）（T5）
3. 两路径差异化渲染（共享核心循环，差异在视觉主题/文本语义）（T5）
4. 难度递增算法（T5）
5. 结局画面（T5 或 T7）
6. Web Audio 合成音效（T7）
7. 动效打磨：命中/失败/重开的反馈动画（T7）

## 风险

| 风险 | 概率 | 影响 | 缓解 |
|---|---|---|---|
| 两路径差异化过大导致工作量翻倍 | 中 | 高 | T4 owner gate 确认：若差异过大则收敛为单路径+另一角色作配角 |
| 移动端触摸延迟影响时机感 | 中 | 高 | 用 Pointer Events（非 Touch Events 延迟 300ms）、testMode 精确测量 |
| Web Audio 合成音色不佳 | 低 | 中 | T7 若合成不佳可引入 CC0 音效资产（须写 ADR） |
| 横竖屏布局差异大 | 低 | 低 | 响应式 Canvas + 相对坐标，UI 元素随短边缩放 |

## 验证计划

- **T5（原型阶段）**：Vitest 单测 judgeTiming 全分支；Playwright testMode 断言游戏循环确定性
- **T6（玩测阶段）**：3-5 人无解说试玩，≥1 份手机端 raw 证据
- **T7（打磨阶段）**：Playwright 截图前后对比，断言颜色数量 ≤4、字体声明、≥3 关键事件有音频触发
- **T8（发布阶段）**：build 产物无绝对路径引用、mobile viewport 断言无 404
