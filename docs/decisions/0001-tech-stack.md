# ADR-0001: 技术栈 — 原生 Canvas+JS + Vite + Vitest + Playwright

- 状态：accepted
- 日期：2026-06-23
- 决策人：用户 + planner

## 背景

本项目是一款个人风格的浏览器小游戏（"守序者与闯入者"），核心机制为"单键时机"。首作目标是在最短时间内做出一个可玩、有手感的网页游戏，而非先学习复杂引擎。技术栈选择需要权衡：开发速度、学习成本、移动端兼容性、长期可维护性。

约束：
- 首款小游戏，范围必须冻结（单键时机 + 浏览器 + PWA）
- 开发者有 Web 开发能力，但无游戏开发经验
- 需要同时支持桌面和手机浏览器
- 不做原生 App 打包/不上架 App Store（本期）

## 决策

采用原生 HTML5 Canvas + 纯 JavaScript（ES2022+）实现游戏逻辑与渲染，Vite 作为构建/开发/预览工具，Vitest 做核心逻辑单测，Playwright 做运行时验证。

具体配置：
- **渲染**：原生 Canvas 2D API（无 WebGL，无引擎）
- **语言**：JavaScript（不引入 TypeScript，除非另写 ADR）
- **构建**：Vite（vanilla JS 模板，`base: './'`）
- **测试**：Vitest（纯函数单测）+ @playwright/test（运行时验证）
- **输入**：键盘 + 触摸统一为"单键按下"事件
- **音频**：Web Audio API 合成（默认不引入外部音频资产）
- **部署**：GitHub Pages + itch.io（均用相对 base `./`）

## 备选方案

| 方案 | 优点 | 缺点 | 为什么没选 |
|---|---|---|---|
| **A：原生 Canvas+JS（我们选的）** | 零学习税、直接控制渲染循环、文件体积小、MDN Games 明确推荐"从纯 JS 入手最利理解 web 游戏" | 需自己管理游戏循环、输入、音频；无内置物理/场景管理 | （选了） |
| B：Phaser | 成熟游戏引擎、内置场景/物理/音频管理、社区大 | 学习税高、文件体积大、对首款小游戏偏重 | 引擎学习税对零基础不友好 |
| C：PixiJS | 高性能 2D 渲染、比 Phaser 轻 | 仍非必需、首作无需高性能渲染、增加了抽象层 | 原生 Canvas 足够 |
| D：Kaboom.js | 轻量、API 简洁 | 官网已声明"不再维护"（2024 年起），存在长期风险 | 维护风险 |
| E：Godot Web 导出 | 完整引擎、可视化编辑 | 对首款小游戏偏重、Web 导出额外复杂度、需要学习 Godot | 过重 |
| F：p5.js | 艺术/创意编程友好、社区活跃 | 面向艺术/玩具型项目，非游戏架构；性能 overhead | 不匹配游戏型项目 |

## 后果

### 正面
- 最快启动：无需学习引擎 API，直接写游戏逻辑
- 完全控制：游戏循环、渲染、输入、音频全部可控，便于打磨手感
- 文件体积小：无引擎 bundle overhead，适合移动端加载
- 纯函数核心逻辑：时机判定等函数天然可单测

### 负面（重要）
- 需自行管理游戏循环（requestAnimationFrame）、状态机、精灵图/动画
- 无内置物理引擎——若原型证明需要物理，需自行实现或引入库（届时须写 ADR）
- 无内置场景/关卡管理——若证明需要，须写 ADR 评估 Phaser 等引擎迁移
- 音频合成工作量：Web Audio API 比引擎内置音频更底层

## 失效条件

以下信号出现时，应重新审视本 ADR：

1. 原型证明需要复杂物理（碰撞、重力、刚体）且自行实现成本过高 → 评估 Phaser Matter.js
2. 原型证明需要场景/镜头/关卡管理系统且自行实现成本过高 → 评估 Phaser Scene 或 PixiJS
3. 原型证明需要 tilemap 且自行解析成本过高 → 评估 Phaser Tilemap
4. 团队新增有 TypeScript 经验的成员且一致认为 TS 收益 > 迁移成本 → 写 ADR 评估 TS 迁移

出现任一信号时，不直接切换技术栈，而是：写新 ADR → 比较迁移成本与收益 → 决策。
